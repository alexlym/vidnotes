# Plan: `dl-summarizer` — deeplearning.ai Course Summarizer CLI

## Context

Build a Python CLI that takes a course URL on learn.deeplearning.ai, authenticates via a config file, walks every lesson, extracts content (video captions + page text), and uses the Claude API to generate per-lesson Markdown summaries saved to an output directory.

## CLI — Minimal Interface

```
dl-summarizer <COURSE_URL>
```

That's it. One argument: the full URL of the course to summarize.

Optional flags:
```
--output-dir DIR     default: ./output
--force              re-summarize lessons that already have a .md file
--dry-run            extract content, skip Claude call, print what was found
```

Credentials and settings live entirely in `config.toml`.

## `config.toml`

```toml
[options]
output_dir = "./output"
```

That's it. No credentials stored anywhere.

- **Auth**: one-time interactive browser login (Playwright opens a visible Chromium window, you log in normally, window closes, cookies saved to `~/.config/dl-summarizer/session.json`)
- **Summaries**: generated by the `claude` CLI using your logged-in Claude account — no API key needed

## Authentication — Key Finding

The auth server (`https://auth.deeplearning.ai`) is a custom OAuth 2.0 / OIDC server:
- Token endpoint: `https://auth.deeplearning.ai/api/auth/oauth2/token`
- Grant types: `authorization_code`, `client_credentials`, `refresh_token`
- **`password` grant is NOT supported** — cannot POST email+password directly with `requests`

**Consequence: Playwright is required for login.** The `authorization_code` flow is browser-redirect based. After login Playwright extracts the session cookies; all subsequent requests (catalog, lesson pages) use plain `requests` with those cookies.

Login flow (interactive, one-time):
1. Launch **visible** (non-headless) Chromium → navigate to `https://auth.deeplearning.ai/login`
2. Print to terminal: `"Browser opened — please log in, then wait..."`
3. Wait for the URL to contain `learn.deeplearning.ai` (user logs in however they want — email/password, Google SSO, etc.)
4. Extract all cookies from the browser context
5. Save to `~/.config/dl-summarizer/session.json` (chmod 600) → close browser
6. Print: `"Session saved."` → continue with the run

On subsequent runs, cookies are loaded silently. If a request returns 401 or redirects to the login page, the tool detects the expired session and repeats the above. No credentials are ever stored.

## One-Time Discovery Step (Transcript Element Only)

DevTools is only needed to find the transcript element selector on a lesson page. This is a one-time lookup:

1. Open a video lesson on learn.deeplearning.ai
2. Click the **"Read Transcript"** button
3. Right-click the transcript text → Inspect
4. Note the CSS selector or `data-testid` of the container element
5. Record it in `config.py` as `TRANSCRIPT_SELECTOR`

This is a 30-second task done once during setup, not during every run.

## Module Responsibilities

### `session.py`
1. `login()` — launch **visible** Chromium, print login prompt, wait for redirect to `learn.deeplearning.ai`, save cookies, close browser
2. `load_session()` → returns a `requests.Session` with saved cookies injected
3. `validate_session(session)` → GET a lightweight authenticated endpoint; if 401/redirect → call `login()` again

### `catalog.py`
- Given a course URL, enumerate all lessons (id, slug, title, url, type)
- Try REST API first; fall back to parsing HTML sidebar/TOC; fall back to Playwright

### `fetcher.py`
- Default: `RequestsFetcher` using the `requests.Session` loaded from saved cookies
- Fallback: `PlaywrightFetcher` if the page is JS-rendered (cookies transferred from session.json)
- `needs_js_rendering(html)` → detect empty SPA shell (body text < 200 chars) → auto-retry with Playwright
- Playwright is already a dependency (needed for auth), so this fallback has no extra install cost

### `extractor.py`
The lesson page has a **"Read Transcript"** button that reveals the full spoken transcript as text in the DOM. This is the primary extraction method — much simpler than the Wistia API.

Extraction strategy (waterfall, stops at first success):
1. **Page transcript element** — fetch lesson page, find the transcript container using `TRANSCRIPT_SELECTOR` from `config.py` (one-time lookup via Inspect on the "Read Transcript" button). If the transcript is toggled via JS (button click required), use Playwright to click it and wait for the element.
2. **Page article/reading text** — for non-video lessons, scrape written content from the main content area
3. **yt-dlp subtitles** — fallback if neither above yields text

### `summarizer.py` — Using the `claude` CLI

Summaries are generated by calling the `claude` CLI as a subprocess. No API key required — uses your logged-in Claude account.

```python
import subprocess

def summarize(lesson_title: str, course_title: str, transcript: str, page_text: str, prompt_template: str) -> str:
    prompt = prompt_template.format(
        course_title=course_title,
        lesson_title=lesson_title,
        transcript=transcript,
        page_text=page_text or "(none)",
    )
    result = subprocess.run(["claude", "-p", prompt], capture_output=True, text=True, check=True)
    return result.stdout.strip()
```

### Prompt Storage — Best Practice

**Store prompts as plain-text files, not in `config.toml`.**

Why:
- Prompts are long, multi-line, and frequently iterated on — TOML multiline strings are awkward to edit
- A `.md` or `.txt` file opens cleanly in any editor with syntax highlighting
- Easy to version-control and diff changes to the prompt independently of code/config
- User can swap prompts without touching `config.toml` at all

Layout:
```
~/.config/dl-summarizer/
├── session.json          # auth cookies (auto-written)
└── prompts/
    └── summarize.md      # default prompt (shipped with the tool, user can edit)
```

The tool ships a `default_prompt.md` bundled inside the package. On first run it copies it to `~/.config/dl-summarizer/prompts/summarize.md`. From then on, the user's copy is used — edits persist across updates.

**Default `summarize.md`:**

```
You are a technical educator writing concise study notes from online course lessons.
Only use information present in the source material. Preserve all technical terms,
formulas, and code exactly as given. Use Markdown throughout.

Course: {course_title}
Lesson: {lesson_title}

## Transcript
{transcript}

## Page Content
{page_text}

---
Summarize using exactly these sections:

### Overview
2–3 sentences: what this lesson covers and why it matters.

### Key Concepts
Bullet list of core ideas, definitions, and techniques.

### Technical Details
Formulas, algorithms, code patterns, hyperparameters.
Use fenced code blocks for all code.

### Practical Takeaways
What to remember and apply in practice.

### Review Questions
3–5 self-test questions to check understanding.
```

**Why this structure works:**
- Fixed sections → consistent notes, easy to compare across lessons
- "Only use information present" → no hallucination
- No chain-of-thought instruction → direct output, no preamble to strip
- Transcript length: if > ~150k chars, truncate and log a warning

## Key Risks

| Risk | Mitigation |
|---|---|
| Auth uses JWT-in-body not session cookie | After DevTools: store token in session.json, send as `Authorization: Bearer` |
| Platform is fully JS-rendered | `fetcher.py` auto-detects empty SPA shell, retries with Playwright |
| Wistia not used | `--dry-run` dumps raw HTML; add new regex patterns to `extractor.py` |
| Rate limiting by deeplearning.ai | `time.sleep(1)` between lesson fetches |
| yt-dlp blocked without auth | `ydlp_cookies_browser = "chrome"` in `config.toml` |
